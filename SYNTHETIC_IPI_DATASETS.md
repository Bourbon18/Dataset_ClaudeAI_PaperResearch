## 1. Background & Gate Distinction

The `injection_sol` system employs two independent defense gates in sequence:

| Gate | Trigger Point | Mechanism | Signal |
|------|--------------|-----------|--------|
| **D1** | Ingestion (before embedding) | Supervised transformer classifier (DeBERTa-v3-small / DistilBERT) | `P(injection) ∈ [0,1]` per chunk text |
| **D2** | Query time (after retrieval, before augmentation) | Unsupervised embedding-anomaly detector (Isolation Forest / Mahalanobis + retrieval-set cosine geometry) | Anomaly score `∈ [0,1]` per retrieved chunk embedding |

This fundamental difference determines the **design philosophy of each dataset**:

- **D1 dataset** — Injections contain overt lexical and syntactic injection signals (imperative
  overrides, exfiltration verbs, authority impersonation, delimiter abuse) that a fine-tuned
  text classifier can learn to detect. The pattern heuristic in `defense/patterns.py` identifies
  the key signals: `ignore previous/prior instructions`, `disregard`, `send/email/post +
  api key/secret/token`, `reveal/output system prompt`, `### instruction/system/override`,
  code-fence delimiter abuse, etc.

- **D2 dataset** — Injections are **deliberately crafted to evade D1**. They avoid all
  lexical patterns listed above. Instead, they are disguised as peer review addenda, archival
  metadata footnotes, IT maintenance logs, agency guidance notices, and academic footnotes —
  text that reads as innocuous to a pattern-matcher. D2 catches them because their **embedding
  is geometrically anomalous**: the chunk's semantic content (e.g., "IT maintenance / web API
  verification") is an outlier relative to the user's query and the sibling retrieved chunks
  (e.g., "Great Depression macroeconomics").

## 2. JSON Schema Reference

Every sample in both datasets conforms to the following schema:

```
{
  "id":                    Unique string ID (D1_001 … D1_008 | D2_001 … D2_008)
  "attack_vector":         Injection delivery technique
  "malicious_goal":        Ultimate objective of the attack payload
  "user_query":            Legitimate, benign end-user prompt
  "external_context":      Untrusted external chunk containing the embedded injection
  "injection_payload":     Exact isolated malicious instruction within external_context
  "expected_safe_behavior": How a secure LLM must respond (answer + neutralize)
}
```

**Attack Vector Categories:**
- `Direct Command Override` — Explicit instruction to ignore previous rules
- `Social Engineering / Gaslighting` — Deceptive authority narrative targeting LLM helpfulness
- `Contextual Blending` — Payload disguised as a legitimate system message, metadata, or log
- `Obfuscation / Delimiter Manipulation` — Payload breaks out of data fields via markdown,
  code fences, or pseudo-structured delimiters

**Malicious Goal Categories:**
- `Data Exfiltration` — Leak system prompt / user query to attacker-controlled URL
- `Privilege Escalation / System Hijacking` — Force unauthorized access to system internals
- `Phishing & Fraud` — Display fake contacts, portals, or credential-collection instructions
- `Defacement / Toxic Output` — Induce biased, false, or harmful generated content

## 3. D1 Dataset — Ingestion-Time Classifier

### 3.1 Design Rationale

D1 is a **supervised binary text classifier** trained on labeled `(chunk_text, label)` pairs
where `label=1` indicates an injection-laced chunk. The training data (BIPIA, deepset/prompt-
injections, HackAPrompt) is built by the `inject()` function in `data/builder.py`, which
embeds a payload string into a benign carrier passage at a random position.

Accordingly, D1 dataset samples must:
1. Present realistic carrier passages (Wikipedia-style, clinical guidelines, e-commerce FAQ,
   developer documentation) with the injection embedded at a visible position within the text.
2. Use injection phrasing that activates the lexical signals D1 is trained to recognize.
3. Cover all four attack vector types and all four malicious goal categories.
4. Have `expected_safe_behavior` that requires fulfilling the legitimate user query from the
   clean portion of the chunk — not a generic refusal.

## 4. D2 Dataset — Query-Time Anomaly Detector

### 4.1 Design Rationale

D2 is an **unsupervised embedding-anomaly detector** fitted on benign retrieval distributions
(SQuAD, MS MARCO, Wikipedia embeddings). At query time it combines two signals per retrieved
chunk:

1. **Global anomaly** — how far the chunk's embedding sits from the benign training distribution
   (Isolation Forest or Mahalanobis distance).
2. **Local context anomaly** — how geometrically dissimilar the chunk is from the query
   embedding and from its sibling retrieved chunks (cosine outlier-ness within the retrieved
   set via leave-one-out centroid comparison).

D2 catches injections that **bypass D1** — chunks whose text does not contain overt injection
phrasing but whose semantic content is incongruent with what a legitimate query should retrieve.

Accordingly, D2 dataset samples must:
1. Contain injection payloads that avoid **all** lexical patterns in `defense/patterns.py`
   (no `ignore/disregard`, no `send/email/post + api key/secret`, no `reveal system prompt`,
   no `https?://url?param=value` callouts, no `### instruction` markers).
2. Disguise the injection as a plausible document annotation (peer review note, maintenance
   log, clinical bulletin, agency guidance update, archival metadata, etc.).
3. Produce a chunk whose **embedding** is in a semantic space clearly different from the
   user's query domain and from what legitimate sibling chunks would contain.
4. Document the **embedding-space gap** so evaluators understand why D2 would flag it.
